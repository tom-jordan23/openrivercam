# ORC-OS Fitment Station (Docker)

*A containerized OpenRiverCam OS running locally, used to try the camera-fit workflow through the real operational UI instead of the raw Python script.*

## What This Is

[ORC-OS](https://github.com/localdevices/ORC-OS) is the OpenRiverCam dashboard — the same software that runs on a deployed Raspberry Pi 5 station. It provides a web UI for uploading videos, setting up camera configurations (GCP clicks, z_0, cross-section), running velocimetry, and reviewing results. We run it locally in Docker for fitment testing before committing to field hardware.

Relationship to our other work:

- **`survey/orc_build_camera_config.py`** — Python script that wraps `pyorc` directly. Fastest iteration on residual analysis.
- **ORC-OS (this doc)** — full operational UI. Closer to what the production station will look like; better for end-to-end sanity checks and for handing a working demo to the team.
- **`survey/ORC_FIT_STRATEGY.md`** — action plan for interpreting fit residuals. Applies regardless of which tool produced the fit.

## Running the Stack

The build uses `docker` aliased to `podman` on this machine. `docker compose` resolves to the external `docker-compose` v2 binary, which works with podman.

```bash
cd rainbow-sensing/orc-os

# first build (downloads base images, builds python + frontend — several minutes)
docker compose up --build -d

# follow logs
docker compose logs -f

# stop
docker compose down

# rebuild after code changes
docker compose up --build -d --force-recreate
```

### Podman VM memory note

The podman machine is configured with 3.7 GB RAM. The `orcapi` container has `mem_limit: 8g` in the compose file — it will be capped at the VM memory ceiling. For interactive fitment this is fine. If velocimetry runs spike and get OOM-killed, bump the VM:

```bash
podman machine stop
podman machine set --memory 8192
podman machine start
```

## Fitment Workflow in the Web UI

1. Open http://localhost:3000.
2. Create a new site (sukabumi).
3. Upload or point to the scene video in `uploads/incoming/`.
4. Start a new camera configuration.
5. Click GCP positions on the video frame; paste the UTM x, y, z for each point from `gcps.csv`.
6. Enter z_0 = 617.065.
7. ORC-OS computes the fit and shows reprojected points and a residual summary.
8. Iterate as per `survey/ORC_FIT_STRATEGY.md` — Sections 1 and 5.

If you prefer the Python script path for faster drop-one iteration, the script reads the same CSVs and prints per-GCP residuals sorted worst-first. See `survey/orc_build_camera_config.py`.

## Local Patches Applied to the Clone

The clone at `rainbow-sensing/orc-os/` has three edits on top of upstream. If you re-clone, apply these again.

### `Dockerfile` — pin setuptools<70 in the PEP 517 build env

Symptom without the patch: `ModuleNotFoundError: No module named 'pkg_resources'` while building rasterio 1.3.11 from sdist.

Root cause: pyopenrivercam 0.9.4 pins `rasterio<1.4.0` for python ≤ 3.12. On linux/arm64 py3.11 there is no prebuilt rasterio 1.3.x wheel, so pip builds from sdist. rasterio 1.3's `setup.py` imports `pkg_resources`, which setuptools ≥ 70 no longer auto-exports. PEP 517 build isolation installs the latest setuptools into a fresh env and the build fails.

Replace the "Install Python dependencies" block in the Dockerfile with:

```dockerfile
RUN echo "setuptools<70" > /tmp/pip-constraints.txt
ENV PIP_CONSTRAINT=/tmp/pip-constraints.txt
RUN pip install --no-cache-dir --upgrade pip "setuptools<70" wheel
RUN pip install --no-cache-dir .
```

`PIP_CONSTRAINT` propagates into build-isolation envs on pip ≥ 23.2, so rasterio's build env also gets setuptools<70.

### `pyproject.toml` — dedupe rasterio constraint

Upstream has two `rasterio` specifiers in the dependencies list (one capped `<=1.3.11` guarded by `python_version < '3.10'`, one unbounded for `>= '3.9'`). pip sometimes resolves both through its backtracking logic and still tries to install the capped version on py3.11. Reducing to a single unbounded `"rasterio"` avoids that path. The real gating constraint (`rasterio<1.4.0`) is still enforced by pyopenrivercam itself.

### `Dockerfile` — pin xarray<2026 in pip-constraints

Symptom without the pin (2026-04-22): video processing runs through frame scanning and PIV, then dies on transect extraction with `TypeError: Passing a Dataset as data_vars to the Dataset constructor is not supported. Use ds.copy() to create a copy of a Dataset.`

Root cause: pyorc 0.9.4's `api/velocimetry.py:224` does `ds_points = xr.Dataset(ds_points, attrs=ds_points.attrs)` — re-wrapping an existing `xarray.Dataset` in a new `Dataset()` constructor. That pattern was legal in older xarray; xarray ≥ 2026.x explicitly rejects it with a `TypeError`. pyorc hasn't yet tracked the xarray API change.

**Fix (applies everywhere — local Docker, station image, any rebuild):** add `xarray<2026` to the pip-constraints file alongside `setuptools<70`. Because `PIP_CONSTRAINT` propagates into build-isolation envs, this caps xarray at install time and the bug never triggers.

Replace the constraints line in the Dockerfile with:

```dockerfile
RUN printf "setuptools<70\nxarray<2026\n" > /tmp/pip-constraints.txt
```

Verified compatible versions: xarray 2024.11.0 through 2025.9.0 all accept `Dataset(existing_dataset)` without complaint. The pin gives pyorc a working xarray without locking it to an ancient release.

Remove the `xarray<2026` line when pyorc lands the `ds.copy()` fix upstream (or when ORC-OS upgrades its pyorc pin past that commit).

## Troubleshooting

- **`http://localhost:3000` times out.** Check `docker compose ps`; dashboard depends on orcapi being healthy (up to 40 s startup grace period).
- **`http://localhost:3001/api/` returns 401.** `ORC_DEV_MODE` was not set to 1, or the env file was not picked up. Stop the stack, confirm the `.env` file, and `docker compose up -d` again.
- **OOM during processing.** Bump the podman machine memory (see "Podman VM memory note" above).
- **Video not visible in the UI.** Confirm the file is under `~/.ORC-OS/uploads/incoming/` on the host, and that ORC_DATA_PATH in `.env` matches that path.
- **Database locked.** SQLite write contention. Stop the stack, try again. If persistent, delete `~/.ORC-OS/orc-os.db` to reset (loses all sites and configs).
